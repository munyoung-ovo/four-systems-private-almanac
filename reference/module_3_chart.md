# 命盘解读

触发：命盘、命格、认识自己、事业、感情、财运、贵人、健康或一件具体人生问题。

## 路由

- 用户问整体或首次建档后看盘：用 `prompts/deep_chart_brief.md` 给快照。
- 用户直接问专题或具体事情：跳过快照，使用短答提示直接回答。
- 用户明确要求详细版、报告或选择 `[详细版/1]`：从 `profiles/_ui_state.json` 解析最近专题，使用 `prompts/deep_chart_long.md`。
- 用户再次选择同一专题时，优先深入现状、未来一年、适合方向、避雷点或具体对象，不重复短答。
- 用户回复 `1`、`详细版`、`展开`、`深度解读` 时，都按 `[详细版/1]` 处理；不要因为上一轮没有完整菜单就丢失最近专题。

## 命盘首屏

整体命盘或首次建档后的快照，先给一条直接、能落到生活的总判断，再给当前阶段的一句人话。若用户是来认识自己或验证命盘，给三条来自不同维度、可明确回答“像/不像”的观察；不要把用户的选择倒灌回盘面结论。

快照结尾保留一行好用的入口：[详细版/1] [感情/2] [事业/3] [具体的事/4]。财运、贵人、健康不必每次都挤成完整菜单，用户可以直接说；后续回答按当前问题保留 2-4 个最相关入口。用户再次选择同一专题时，改为深入现状、未来一年、适合方向、避雷点或具体对象，不重复第一段。

## 执行

1. 读取当前档案及审计；资料变化或审计缺失时才重排/重审。
2. 按 `reference/response_boundary.md` 先锁定盘面结论，再使用对话安排建议。
3. 按 `reference/fusion_protocol.md` 提炼多系统共同主题和冲突，不把四套系统并排堆给用户。
4. 用 `engines.ui_state.set_chart_topic(...)` 保存专题或具体问题。

用户明确表示“这一段很像”“想继续看这个方向”时，可以把该方向保存为 `concerns` 或最近专题，用来决定下一轮优先解释什么；它只是兴趣和沟通上下文，绝不进入命盘证据、改写盘面结论、改变系统权重或制造迎合性验证。若用户反馈与结论不符，仍按出生资料、历法、时间与地点边界复核。

## 整体核验

只有用户要求认识自己、完整命盘、验证准确度，或首次建档后没有具体问题时，才给 3 条盘面核验。每条必须来自不同维度，并能被“像/不像”明确验证；用户已经问具体事情时直接回答，不先做性格测试。

用户反馈只用于检查出生资料和解释是否覆盖正确层次，不得进入盘面证据或改变系统权重。若都不像，依次复核出生日期、历法、时间来源、地点和真太阳时边界；资料无误时承认当前断法解释力不足，不把责任自动归给出生时辰。

## 具体处境

只有“主动/被迫”或“已决定/仍在评估”的差异会实质改变建议时，才追问其中最关键的一项。已能从问题判断时不要重复提问。回答时分别说明：本命惯性如何影响选择、当前阶段提供什么条件、具体时间是否适合推进，以及哪一步最可逆。

## 降级

- 出生时辰不确定：省略时柱、命宫细节、上升/天顶及精细分盘结论，只保留不依赖时辰的证据。
- 特殊格局或喜用置信不足：不以普通喜忌给强建议。
- `high_risk`：先核对出生资料或材料冲突，再写详细版。

短答和标准详细答结尾只保留 2-4 个相关入口，其中命盘类回答默认包含 [详细版/1]。整体快照优先使用 [详细版/1] [感情/2] [事业/3] [具体的事/4]；长文写入 outputs，聊天给摘要与路径；入口不写入文档。
