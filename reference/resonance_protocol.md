# 共振冲突协议（Resonance Protocol）

## 四类冲突分类法

### 1. 不同时间尺度（timeframe）
- **现象**：紫微论年/月级趋势说"好"，八字论日级节律说"忌"
- **解读**：两者不矛盾——大势向好，但今日具体节点不佳
- **建议**：听八字（精度更高）；紫微提供"这个月整体方向"的背景色

### 2. 不同领域（domain）
- **现象**：一个系统判断某类事好，另一个系统判断的是不同属性
- **解读**：系统关注维度不同，不构成真正对立
- **建议**：按 act 最相关的系统权重排序回答

### 3. 真正对立（opposite）
- **现象**：两个及以上系统在同一维度对同一事项给出相反判断
- **解读**：真实信号冲突，不可掩盖
- **建议**：透明呈现冲突；保守建议（避免启动大事）；提示用户自主判断
- **条件**：必须同时满足①同维度②同事项③≥2系统对立

### 4. 颗粒度差异（granularity）
- **现象**：粗粒度系统（紫微论年）favor，细粒度系统（Nakshatra 论时）avoid
- **解读**：在更细的颗粒度上可以优化，而不是完全否定
- **建议**：在大框架 favor 的前提下，用细粒度系统找具体吉时

---

## 打分规则

```
resonance_strength = count(favor votes)  # 0-4，纯计数，不加权平均

is_destined_moment = (resonance_strength == 4)  # 4/4 才触发，宁缺毋滥
```

### 强度解读

| strength | 解读 | 建议行动 |
|----------|------|---------|
| 4/4 | 命定时刻·四系统共振 | 主动出击，窗口珍贵 |
| 3/4 | 强吉日·三系统认可 | 推荐，说明一个保留系统 |
| 2/4 | 中性·各半 | 可行，但风险自担 |
| 1/4 | 弱·仅一系统认可 | 建议另选日期 |
| 0/4 | 四系统均不支持 | 明确建议延期 |

---

## 冲突呈现格式（供 LLM 用 electional.md 输出）

```
[系统A] favor：[具体原因]
[系统B] avoid：[具体原因]
冲突类型：[timeframe / domain / opposite / granularity]
建议：[一句话结论]
```

---

## 禁止事项

- 禁止对医疗、手术时机给出"吉/凶"判断——只能说"传统上认为XX日较平稳，但请遵医嘱"
- 禁止在"真正对立"冲突下强行给出正面结论
- 禁止把颗粒度差异归入真正对立（夸大矛盾）
