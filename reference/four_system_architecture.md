# 四系统与模型路由架构

## 总体分层

四套系统共享同一条流水线，但保留各自的计算方法和适用尺度：

```text
出生资料与目标日期
→ 输入规范化与边界审计
→ 四套确定性排盘
→ 各系统证据层
→ 专题证据包
→ 可靠度感知融合
→ 模型分级表达
```

计算层不使用对话。证据层只抽取程序已给字段。融合层处理领域、时间尺度、强度、置信度和冲突。生成模型只负责综合解释与行动表达。

## 统一系统契约

每套系统都应提供：

- `calculation_profile`：历法、时间精度、坐标/时区、置信度、证据等级和警告。
- `degraded`：是否整体降级。
- 本命层：不随咨询日期改变的结构。
- 时间层：明确绑定 `target_date` 的阶段或流运。
- 解释索引：按主题快速取得相关宫位、行星、星曜或阶段数据。
- `validation`：数量、边界和内部一致性检查。

进入模型前使用 `engines.evidence_packet.build_chart_packet(profile, topic)`，只传当前专题需要的字段。`engines.chart_signals.build_topic_signals(...)` 将可确定的宫位、阶段、八分图和行运条件转换为带来源字段的专题信号。档案的 `calculated_for` 与问题日期不一致时，先使用 `engines.build_profile.refresh_profile_for_date(profile, target_date)` 刷新时间层。

专题信号规则属于保守解释层，不冒充原始排盘：每条都带 `source_fields`、`scope`、`strength` 和 `confidence`。简单映射规则设置信心上限；缺少足够数据时返回 `insufficient`，不要求模型补断。

## 八字

职责：本命底色、日主承受方式、格局、喜忌可靠性、大运流年和日课互动。

架构：

1. 本命层：四柱、十神、纳音、格局与真太阳时修正。
2. 强弱层：旺衰分数、分解、边界置信度和特殊格局识别。
3. 时间层：按目标年份定位当前大运与流年，不把建档年份永久当作当前年份。
4. 证据层：强弱置信度不足、时辰未知或特殊格局存在时，自动移除普通喜用神强结论。
5. 日课层：与具体日期的冲合、贵人、驿马、桃花等只由每日引擎触发。

质量门：四柱与换日/时辰边界可追溯；喜用神必须通过可靠性门槛；阶段结论必须标注目标年份。

## 紫微

职责：人生领域、宫位结构、大限/流年/流月/流日主题及四化落宫。

架构：

1. 本命层：命宫、身宫、五行局和十二宫。
2. 宫位关系层：空宫借对宫、三方四正、吉曜与压力星标签。
3. 专题索引：感情、事业、财富、健康、贵人直接映射相关宫位，不按数组位置猜宫。
4. 时间层：所有流限绑定 `target_date`，四化按宫与类型建立索引。
5. 降级层：时辰未知时不使用命身宫和精细流限作强判断，只保留可确认的背景信息。

质量门：十二宫完整命名；空宫借星来源明确；专题结论必须来自对应宫位；年/月背景不能冒充日级择时。

## 印占

职责：月宿本能、Vimshottari 阶段、Lagna/宫主、分盘、Ashtakavarga 和日时级择时。

架构：

1. 本命层：Lahiri 黄道、月宿/Pada、Lagna、行星、宫主和尊贵度。
2. 分盘层：基础分盘与 vargottama，仅在出生时间精确时启用。
3. 阶段层：Mahadasha/Antardasha 必须定位到 `target_date`。
4. 强度层：SAV/BAV 提供可验证点数；未配置的强度精算必须明确 `available=false`。
5. 择时层：月宿、Tithi、Yoga、Karana 与具体事项映射，保持日/时尺度。

质量门：节点模式和 Lahiri 参数固定记录；Rahu/Ketu 对冲、SAV 总数等校验通过；不能用未配置的强度字段写强弱精算结论。

## 西占

职责：本命心理结构、宫位主题、行星尊贵度、相位和天/周级行运触发。

架构：

1. 本命层：热带黄道行星、速度、逆行和主要相位。
2. 宫位层：Placidus 宫位、上升与天顶，仅在时间和地点精确时启用。
3. 传统层：守护、尊贵度、日夜盘、月相与月亮空亡。
4. 行运层：目标日期行星触发本命点，保留逼近/分离与逆行状态。
5. 专题索引：按感情、事业、财富、健康和贵人选择相关行星与相位，避免传入全盘冗余。

质量门：宫头数量与行星数量校验；无可靠宫位时自动去掉上升/天顶/宫位结论；分离相位只作余波，空亡月对启动事项降权。

## 四系统融合

每条系统证据统一包含：`signal`、`scope`、`domain`、`strength`、`confidence`、`weight`、`effective_weight` 和 `reason`。

```text
effective_weight = 事项基础权重 × 信号强度 × 数据置信度
```

先过滤领域不匹配与降级弱信号，再计算方向。票数只表示收敛度，不直接代表可靠度。高置信反对不能被多个低置信支持淹没；低置信四票同向也不能自动升级为强结论。

冲突依次判断：时间尺度、领域、层次、强度。只有同领域、同时间尺度且强度接近时，才算真实对立并升级到高推理模型。

命盘专题使用 `chart_signals.synthesis`，具体日期使用 `fusion`。两者都只综合可靠方向；`mixed` 表示主题被激活但吉凶未定，不计入支持或压力票。

## 模型路由

调用 `engines.task_router.route_task(...)`，只返回供应商无关的执行档位：

| 档位 | 任务 | 上下文 |
| --- | --- | --- |
| `deterministic` | 建档、档案操作、日历文件生成 | 不调用模型或只模板化 |
| `low_cost` | 菜单、每日短签、文件交付、档案摘要 | 小型证据包 |
| `standard` | 命盘短答、普通择日、合盘短答 | 专题证据包 |
| `reasoning` | 四系统真实冲突、资料高风险、完整长文、复杂合盘、高风险边界 | 扩展证据与审计信息 |

缺少关键输入时优先由低成本模型问一个澄清问题，不把不完整任务升级给高成本模型。长报告、真实冲突和资料完整性风险才升级；文风润色不是升级理由。

## 运行纪律

- 路由在读取大上下文之前执行。
- 低成本模型只接收专题证据包，不接收完整历史对话和整张原始盘。
- 高推理模型输出的仍是结构化结论；最终短文可降回低成本模型润色，但不得改动结论字段。
- 所有模型输出都经过结构校验；失败时重试结构，不重新排盘。
- 记录档位、原因、输入摘要哈希和验证结果，不记录私人原始对话。
