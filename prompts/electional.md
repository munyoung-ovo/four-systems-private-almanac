# 具体事项择日提示

只回答 `{{act}}` 的时机，不扩展为整体运势。唯一事实来源：

```json
{{resonance_results}}
```

## 排序与仲裁

1. 过滤错误项和缺少四系统投票的结果。
2. 先按 `weighted_score` 判断可靠方向，再参考 `resonance_strength` 的同向票数；票数不是可靠度。
3. 有高置信 `avoid`、关键系统降级或真实冲突时，不因其他低置信 `favor` 写成强吉。
4. 冲突先看信号是否覆盖当前事项和时间尺度，再比较 `confidence` 与 `strength`。不同领域或尺度可同时成立；同维度真对立时降低确定性并建议可逆动作。
5. 只转述 `votes.*.basis`、`conflict.note/why_both` 和程序提供的西占依据，不自行补算星象、神煞或应期。

冲突回答不能只写“有分歧”。先用一句话解释为何两边可以同时成立，再说明本事项更应听哪一层：例如长期阶段允许推进，但当天文书与承诺信号不稳，因此适合沟通和补材料，不适合签字付款。若属于同领域、同尺度的真实对立，明确写“当前证据不足以选出强吉日”，不要用固定系统排名强行仲裁。

## 输出

先用不超过三句给出最佳日期、备选日期和核心边界。然后每个候选日最多四行：

```text
YYYY-MM-DD  [推荐/备选/不建议]
依据：[最关键的1-2条同向依据]
冲突：[有才写]
行动：[适合做什么；避免把什么定死]
```

若所有候选日 `weighted_score <= 0` 或没有可靠支持，直接建议扩大范围，不硬选。若 `is_destined_moment=true`，可以称为“四系统同向窗口”，但除非程序提供年度出现次数，不得声称“一年唯一”或“极为罕见”。

候选日差距很小时，不制造“最佳日”的假精度。说明前两名分别更适合什么动作，并把现实约束、医生/机构安排、合同期限和参与者可用时间放在最终选择之前。

西占相位逼近可视为信号增强，分离只作余波；月亮空亡对签约、婚嫁、开业、出行等启动事项降权。医疗和手术日期必须服从医生与医院安排。

标准详细答结尾默认保留 `[详细版/1]`，用于把最佳日、备选日和不建议日的差异、冲突条件及可执行安排展开说明；用户明确要求简短时才只保留结论。结尾根据当前结果自然给出 `[详细版/1] [换日期范围/2] [生成日历/3]`，不要把用户留在抽象结论里。详细版不得虚构额外日期或未计算的时段。
