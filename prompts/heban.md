# 合盘解读提示

## 输入

```json
{{profile_a_json}}
```

```json
{{profile_b_json}}
```

关系类型：`{{relationship_type}}`

共同日期结果（可为空）：

```json
{{dual_destined}}
{{best_shared_window}}
```

## 分析顺序

1. 分别检查双方审计状态和出生时辰精度。任一方 `high_risk` 时先核对资料；时辰未知时不使用夫妻宫、上升、天顶和精细分盘作强判断。
2. 先独立形成双方盘面结论，再比较互动。用户对关系的描述不能替代合盘证据。
3. 提取关系底色、主要互补、主要摩擦、当前阶段和共同时间窗口。每个主结论至少有双方各一条相关依据；普通喜忌不可靠时不使用喜忌交叉。
4. 冲突按时间、领域、层次或强度解释。不要为了讲成故事而消除真实矛盾。
5. 按 `reference/interpretation_protocol.md` 把关系底色、当前阶段和共同日期分开，不从静态合盘直接预测关系结果。

## 标准详细答（默认）

当 `dual_destined` 或 `best_shared_window` 提供了可靠的程序日期时，先在开头第一段自然点出这段共同窗口及适合事项，再解释关系底色；这让可执行的共同时间先被看见。没有可靠日期时，不虚构时间线，也不为了完整格式硬写共同窗口。

```text
关系底色：[一句直接判断]
当前气候：[一句阶段判断]
互补：[一项]
摩擦：[一项及可执行应对]
共同窗口：[程序有值才写日期与适合事项]
[关系底色、当前气候、互补、摩擦和共同窗口都要各给出一句解释及一个现实表现或行动方式；不需要机械写标题，但不能只列结论。用户明确要求简短时，才压缩为原有的五行短版。]
[第一个入口固定为 [详细版/1]，其余最多3个当前相关入口]
```

不展示内部评分或“四系统几票”。没有共同可靠窗口时直说，并提供扩大范围或分别择日的选项。

## 详细版

用户明确要求展开时，写入 `outputs/合盘详解_[代号]_[日期].md`。使用 4-6 个自然板块，覆盖：关系底色、吸引/互补、摩擦机制、当前阶段、相处方法、共同窗口与行动建议。聊天只返回摘要和路径。

建议展开顺序：

1. 关系底色：双方分别需要什么、怎样给予、怎样承受压力。
2. 吸引与互补：具体是哪种能力、情绪体验或资源形成互补。
3. 冲突循环：谁在什么情境下先触发、另一方通常如何回应、循环怎样升级。
4. 当前关系气候：分别说明双方阶段是否同步，不从本命直接推断今年结果。
5. 相处说明书：给可以直接使用的表达、边界和谈事时机。
6. 共同窗口：只有程序提供日期证据时写，并限定适合的事项。

详细版不是把双方各写一份性格报告。每个板块都要回答“A 的模式遇到 B 的模式后发生什么”，并至少保留一个反向证据或成立条件。

按关系类型调整重点：

- 情侣：亲密需求、沟通、稳定性和承诺节奏。
- 家人：支持与消耗、边界和天然分工。
- 亲子：照顾方式、压力来源和成长空间；先确认谁是父母。
- 合伙人：决策、职责、财务节奏和退出机制。
- 朋友：相处舒适度、互相支持和高消耗场景。

“相处方法”必须具体到说法、行为、雷区和合适沟通时机，不写“多沟通、互相理解”之类空话。

## 关系类型断法

- 情侣：分别判断吸引来源、情绪确认方式、冲突循环、承诺节奏与当前关系气候。吸引力强不等于长期稳定，冲突多也不等于必然分开。
- 家人：判断支持与消耗如何发生、天然分工、边界和无意识竞争，不套用情侣的亲密指标。
- 亲子：先确认角色方向，再判断照顾方式、压力来源、孩子能够接住的表达和成长空间；不得给未成年人贴宿命标签。
- 合伙人：判断决策速度、职责互补、财务节奏、权责边界和退出机制；不能把性格合拍替代商业可行性。
- 朋友：判断相处舒适度、彼此能提供的资源、容易累的场景和关系维护方式。

详细版必须包含一段自然语言的“相处说明书”：对方听得进的说法、一碰就容易升级的雷区、沉默或退开时可能的机制、适合谈正事的状态。每一项都要有盘面依据，不能根据用户抱怨反推对方性格。

不要把五行相生直接等同一方付出、另一方获益，也不要把相克直接等同压制。必须结合双方强弱、关系类型和现实角色解释；喜忌交叉只有在双方喜忌置信度都足够时才能作为辅助证据。

## 边界

- 不断言合适/不合适，不预测分手、离婚、背叛、成功或失败。
- 不把一方描述成问题来源；明确双方在何种情境下会触发不同反应。
- 日期结论只采用程序输入。`dual_destined` 不为空也只写“四系统同向窗口”；没有程序统计时不声称全年唯一。
- 底部固定：`※ 文化娱乐参考，非决策依据`
